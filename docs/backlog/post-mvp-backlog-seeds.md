# Post-MVP Backlog Seeds

## Purpose
Seed executable post-MVP work for the completed governance-kernel MVP so the repository can move from a proven tracer bullet into a durable, operable product lifecycle.

## Current Baseline
- `Phase 0` through `Phase 4` are complete through `GAP-017`.
- The current repository contains contract primitives, repo verification tooling, sample repo profiles, a sample control pack, and a scripted read-only trial path.
- Missing from the product surface are real build artifacts, real doctor/hotspot checks, durable workflow services, persistent workers, object storage, richer replay tooling, bounded team governance, and non-Codex adapter certification.

## P5 Governance Maturity
1. Formalize the clarification protocol and escalation evidence model.
2. Add a control rollout matrix and promotion policy.
3. Add compatibility gating and repo admission signals for repos and adapters.
4. Add evidence templates and completeness checking.
5. Extend the control registry with lifecycle metadata and recurring review semantics.

## P6 Productized Runtime Substrate
1. Add buildable package metadata and replace the remaining `build` `gate_na`.
2. Add a real doctor/hotspot command and replace the remaining `hotspot` `gate_na`.
3. Add task persistence and a durable workflow service boundary.
4. Add execution workers, managed workspaces, and artifact/object storage.
5. Connect quick/full verification and delivery artifacts to real task runs.

## P7 Assurance And Operational Scale
1. Add a failure replay catalog and failure-signature normalization.
2. Add tracked-file and artifact classification policy for risky outputs.
3. Add supply-chain and external baseline observation controls.
4. Expose replay, rollout, and evidence quality through operator tooling.

## P8 Team And Adapter Expansion
1. Add a bounded team approval and delegation model.
2. Add certification for at least two non-Codex adapter shapes.
3. Add adapter certification tests for degradation behavior and evidence adequacy.
4. Keep repo reuse grounded in profiles and compatibility signals rather than source mirroring.

## P9 Selective Advanced Modules
1. Add an optional evidence-aware memory layer that is never the fact source.
2. Define guarded subagent or multi-agent execution policy.
3. Define an optional release or deployment governance module that remains non-default.

## Explicit Non-Goals
- multi-repo distribution hub behavior
- source mirroring or backflow sync
- skill marketplace or promotion lifecycle
- memory-first product architecture
- default multi-agent orchestration
- upstream agent UX replacement
- deployment automation as the default completion path
