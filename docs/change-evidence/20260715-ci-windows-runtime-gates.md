- cause: both Verify jobs ran on `ubuntu-latest`, while the executed runtime contract explicitly checks the Windows process environment and uses executable `.cmd` fixtures. The Linux runner therefore produced nine inherited runtime test-file failures, including `PermissionError` for `fake-dotnet.cmd`.
- inherited proof: default-branch run `29394031470` at exact main SHA `4fd39cd0` and governance PR run `29403150644` at `9be5746a` both report `Completed 104 test files ... failures=9` with the same `.cmd` permission failure.
- change: move only `Repository integrity` and `Release preflight` to the supported `windows-latest` GitHub-hosted runner. The independent target-rule matrix remains on Ubuntu.
- hosted attempt 1: PR run `29404496213` proved both jobs selected Windows, then failed during checkout before tests because Git could not create the repository's deeply archived path (`Filename too long`).
- checkout repair: run `git config --global core.longpaths true` in each fresh job before `actions/checkout`; no repository content is excluded from verification.
- hosted attempt 2: PR run `29413222522` passed long-path checkout and all Windows-specific `.cmd`, doctor, guard, and operator paths. The remaining failures read `Path.home()`, Codex/Claude local status, and the Claude workload probe from a clean runner with no managed global copies or local host configuration. `test_runtime_evolution` also indexed a developer-residue `.runtime` artifact that is absent in a clean checkout.
- follow-up cause: the host-feedback boundary mixed deterministic repo assertions with live machine state, while selector/evolution/certification/recovery dynamically reloaded that module and bypassed local test mocks. The tests therefore encoded the developer machine as an undeclared prerequisite.
- deterministic repair: production continues to use live host state unless `GACR_HOST_FEEDBACK_FIXTURE` is explicitly set. CI points that variable to a strict repository-owned JSON fixture; paths outside `tests/fixtures/host-feedback`, malformed JSON, missing/extra keys, wrong types, and any acceptance scope other than `test_only_not_hosted` fail closed.
- claim boundary: fixture-derived output records `mode=test_fixture`, `acceptance_scope=test_only_not_hosted`, `hosted_acceptance=false`, fixture ref, and SHA-256. Selector/evolution/certification/recovery preserve that provenance; fixture success is never hosted/manual acceptance.
- hosted attempt 3: run `29417375221` at exact head `8a8d00c28493e8053e9bc79c79bcfd0cc823bc77` proved the fixture repair: Repository integrity completed all 104 Runtime test files with `failures=0`. It then correctly failed global sync because an empty GitHub runner had no managed Codex/Claude copies. Release preflight independently failed one Runtime file because that job had not installed `core.hooksPath=.githooks` before doctor. These are distinct clean-runner setup causes, not a repeat of the host-state fixture cause.
- setup repair: both jobs now use the existing family verifier and sync tool in the ephemeral runner profile: family audit -> dry-run -> apply only when changed -> backup assertion for any updated target -> zero-drift fresh-process check. Release preflight also runs the same existing repo-hook installer as Repository integrity. No auth, provider, secret, MCP, account, or persistent user profile is created or modified.
- hosted attempt 4: run `29419492864` at exact head `0e56ebedb5d150d1f58fcb2654c256792acffa6d` proved both setup steps: hook installation and ephemeral family/dry-run/apply/zero-drift sync succeeded in both jobs. Runtime then failed only `test_policy_tool_credential_audit` and its dependent `test_governance_hub_certification` because the rule-only `.codex/.claude` directories were misclassified as active security configuration.
- rule-only repair: local credential-policy audit activation now requires a supported config, settings, or MCP surface. Managed `AGENTS.md/CLAUDE.md` copies alone return `platform_na`; once any supported surface exists, the existing managed-host fail-closed checks remain unchanged. No fake auth/provider/settings or weaker audit result is introduced.
- hosted attempt 5: run `29421759701` at exact head `27d0be2a3a0556707723c9a618f0e9b9ff22c6eb` passed setup, all 104 Runtime files, policy/tool credential audit, and governance certification in both jobs. Contract then reached `verify-target-project-rules.py` and both jobs failed only while printing the completed JSON result because Windows hosted Python 3.14 selected `cp1252` for stdout and the result contains non-ASCII repository metadata.
- output-encoding repair: target-project audit CLI JSON now uses ASCII-safe JSON escaping. Decoded JSON values, verifier status, paths, findings, and fail-closed behavior are unchanged; the transport is safe under both UTF-8 and legacy Windows code pages.
